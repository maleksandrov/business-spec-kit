# /bk.execute [type] — Generate Execution Assets

## Purpose

Generate a specific execution asset. Each asset is written for an external audience and is reviewable standalone.

---

## Supported Types

| Command | Output file | Description |
|---------|-------------|-------------|
| `/bk.execute pitch` | `execute/pitch.md` | Full pitch narrative |
| `/bk.execute proposal` | `execute/proposal.md` | Formal proposal with scope and terms |
| `/bk.execute email` | `execute/email.md` | Outreach or follow-up email sequence |
| `/bk.execute deck` | `execute/deck-outline.md` | Slide-by-slide deck structure with speaker notes |
| `/bk.execute gtm` | `execute/gtm.md` | Go-to-market motion: channels, messaging, sequencing |
| `/bk.execute partner` | `execute/partner-brief.md` | Partnership brief for a prospective partner |
| `/bk.execute objections` | `execute/objections.md` | Objection map with anticipated pushback and responses |

---

## Pre-conditions — CHECK BEFORE PROCEEDING

### Check 1: Asset type is valid

If the type is not in the table above:
> "Unknown asset type '[type]'. Supported types: pitch, proposal, email, deck, gtm, partner, objections."

Stop.

### Check 2: plan.md exists

If `plan.md` does not exist:
> "No plan found. Run `/bk.plan` first."

Stop.

**Note:** `plan.md` does NOT need `Status: Approved` to generate execute assets. Drafting assets in parallel with plan review is allowed.

---

## Behavior

**Step 1.** If critical information is missing and needed for this specific asset type, ask for it before generating. Examples:
- For `proposal`: recipient name and company (ask if not in brief)
- For `email`: recipient role description (ask if target is vague in brief)
- For `deck`: intended format and slide count (ask if not specified)

**Step 2.** Read and internalise:
- `brief.md` — objective, target, constraints, success metrics
- `strategy.md` — strategic approach, key assumptions, risks
- `plan.md` — phases and tasks (for context on what has been planned)
- `.businesskit/constitution.md` — tone, non-negotiables, brand voice
- `.businesskit/glossary.md` — correct terminology

**Step 3.** Generate the requested asset using the appropriate template in `execute/`.

**Step 4.** Every asset must satisfy all of the following:
- Opens with a `## Context` block referencing the `brief.md` objective
- Written for an **external audience** — no internal shorthand, no references to internal documents
- Has `## Version` and `## Status` lines at the top
- Reviewable standalone — a reader with no access to other files must be able to understand it
- Tone and terminology consistent with `.businesskit/constitution.md` and `.businesskit/glossary.md`
- All `{{ placeholder }}` values replaced with actual, specific content

**Step 5.** Write to `execute/[asset].md`.

**Step 6.** Output:
> "[Asset type] drafted at `execute/[asset].md`. Review and edit before use."

---

## Rules

- **Do NOT use placeholder syntax in the output.** Every `{{ }}` must be replaced with real content derived from the brief, strategy, and plan.
- **Do NOT produce generic content.** The asset must be specific to this project's objective, target, and context. Generic outputs fail on contact with real recipients.
- **If a required piece of information is missing** (e.g. no company name for a proposal), ask for it — do not invent it.
- Multiple asset types can be generated independently. Each is a separate invocation.
- An existing `execute/` file can be regenerated by invoking the command again. Confirm with the user before overwriting.

## Output Format

Write to the appropriate file in `execute/`. All placeholder syntax replaced. HTML comments removed. `Status: Draft`. File is valid Markdown.
